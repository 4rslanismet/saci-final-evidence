METHODOLOGY

How SACI is calculated and interpreted

This page explains the SACI measurement model, including CWLC, CAC, MDC, CTIC and TF formulas, evaluation design, assumptions, validation checks and interpretation boundaries.

Methodology at a glance

SACI first defines the evaluation scope and then builds the expected visibility relations in a graph. Each relation is interpreted as observed, missing or out of scope. Metrics are calculated from these relations; non-applicable metrics are treated as N/A and the final score is produced only from the normalized weights of active metrics.

The goal is not to reduce security visibility to a single alert count. The method shows how log source status, detection controls, MITRE mapping, CTI enrichment and freshness jointly become an evidence-based result.

  • 1. Scope is declared: expected asset, log source, control, MITRE technique and CTI relations are defined.
  • 2. The graph is built: expected visibility relations are modeled as edges.
  • 3. Evidence is searched: log, alert, rule, enrichment and coverage outputs are used to validate relations.
  • 4. Metrics are calculated: CWLC, CAC, MDC, CTIC and TF are produced from the relevant evidence relations.
  • 5. The score is normalized: N/A metrics are removed and final SACI is calculated from active weights.
  • 6. The explanation is produced: the explanation layer does not change the score; it makes deterministic outputs readable.

Assumptions

SACI is not designed as a system that discovers the entire unknown attack surface by itself. For the model to work correctly, the evaluation scope must be declared in advance. Critical assets, expected log sources, enabled detection controls, in-scope MITRE techniques and CTI enrichment expectations must be explicit.

This assumption also defines the boundary of the study. SACI does not claim that there are no unknown gaps in the environment. It makes a narrower and measurable claim: are the expected visibility relations in the declared scope validated by evidence?

  • Declared scope: the defined set of assets, log sources, controls, MITRE techniques and CTI relations included in the evaluation.
  • Expected relation: a relation that is expected to close in the graph within that scope.
  • Evidence source: a log, alert, rule, enrichment or coverage output that validates the relation.
  • Boundary: a relation outside the scope is not counted as missing; it is treated as a separate inventory or scope issue.

Core idea

SACI asks a simple question: have the expected visibility relations in the declared evaluation scope really closed? This relation is not limited to “did a log arrive?”. The chain from asset to log source, log source to Wazuh, detection control to rule, rule to MITRE technique and CTI finding to alert is evaluated together.

Therefore, SACI is not a “is the organization secure?” score. It provides a narrower but more measurable claim: what portion of expected visibility relations can be validated within the declared scope?

SACI = weighted average of active metrics
SACI = Σ(metric_score × metric_weight) / Σ(active_metric_weight)

The critical point is that every metric is not forced into every scenario. If a metric is truly not applicable for a scenario, it is treated as N/A. The final score is then normalized using only the active metric weights.

What do the metrics measure and how are they calculated?

Each SACI metric measures a different part of the visibility chain. A log source may be active, but it may not feed a detection control. A control may be defined, but not validated through the expected Wazuh rule or MITRE ATT&CK technique. CTI may contain IOC information, but if that information does not turn into an alert, operational visibility is still incomplete.

SACI is therefore not a single “alert count”. Each metric is calculated from its own evidence base and then active metrics are combined through weighted normalization.

CWLC

Criticality-weighted log coverage

CWLC measures whether critical assets are actually monitored through the expected log sources. Assets are not treated equally; losing visibility from a high-criticality system such as a domain controller affects the score more.

La = Σs(was × oas) / Σswas
CWLC = 100 × Σa(ca × La) / Σaca

Evidence base: asset inventory, expected log sources, log_source_status and asset → log_source graph relations.

CAC

Control / alert coverage

CAC measures whether declared detection controls are connected to the rule and alert chain that produces operational evidence. A control being listed is not sufficient; the control must close through Wazuh rule, alert and evidence relations.

CAC = 100 × Σ(weight × seen × enabled) / Σ(weight × enabled)

Evidence base: control_coverage, detection_controls, Wazuh rule observations and control → rule / rule → alert graph relations.

MDC

MITRE ATT&CK detection coverage

MDC measures whether in-scope MITRE ATT&CK techniques are covered by detection controls. The aim is to show not only that rules run, but also which attack techniques they make visible.

MDC = 100 × covered MITRE techniques / MITRE techniques in declared scope

Evidence base: mitre_coverage, detection-control mappings and control → MITRE or rule → MITRE graph relations.

CTIC

CTI / MISP enrichment coverage

CTIC measures the closure of lookup_executed, MISP hit, Wazuh alert and MITRE mapping fields for synthetic IOC records. Since lookup_executed may be initialized from expected_lookup=1, the first quarter is not independent proof of query execution and the four fields alone do not establish causality.

CTIC = 100 × Σ(lookup_flag + misp_hit + wazuh_alert + mapped_to_mitre) / (4 × IOC count)

Evidence base: ctic_coverage, MISP enrichment outputs, IOC matches and IOC → MISP → alert → MITRE graph relations.

TF

Telemetry freshness

TF scores the distance between the scorer runtime and the newest last_seen value across log-source rows. It is not a per-source freshness average; a globally recent event can keep TF high even if some sources are older.

TF = 100 if newest age ≤60 min; 80 if ≤1440 min; 50 if older; 0 if no event

Evidence base: log_source_status, last_seen timestamps, freshness thresholds and active log-source relations.

Worked example

The examples clarify weighted scoring and N/A normalization. They are illustrative, not the final laboratory result. As the scenario changes, some metrics remain active and some become out of scope. Out-of-scope metrics are not set to zero; they are removed from the active-weight denominator.

Decision rules

The observed or missing status of a relation is not arbitrary. Each relation is based on a predefined expectation and an evidence type that can validate it. If the expected relation is validated by evidence, it becomes observed. If it is declared but no expected evidence is found, it remains missing. A relation outside the declared scope is not converted into missing.

Observed decisionThe expected relation is validated by log, alert, rule, enrichment or coverage output.
Missing decisionThe relation is expected within the declared scope, but the required evidence was not produced.
N/A decisionThe relation or metric is outside the scenario scope and is not interpreted as success or failure.
Reason-code decisionThe reason code explains whether the missing relation comes from the log-source, control, MITRE, CTI or freshness layer.

Data-to-metric mapping

Every metric is grounded in specific files and graph relations. This mapping makes it clear where the score comes from and which evidence layer supports each component.

Asset inventory + log source status → CWLCCalculates whether critical assets are monitored through expected log sources.
Detection controls + Wazuh rules → CACShows whether controls are linked to expected rule and alert relations.
MITRE coverage → MDCValidates whether controls and rules cover declared ATT&CK techniques.
MISP enrichment + IOC matches → CTICEvaluates closure of lookup, MISP hit, alert and MITRE mapping flags.
Last seen timestamps + freshness threshold → TFCalculates whether telemetry is still recent enough to support operational visibility.
saci_nodes + saci_edges → graph closureCombines expected, observed and missing relations in a shared evidence graph.

Weight sensitivity

SACI weights differ because some visibility layers are operationally more foundational than others. Log coverage and control/alert coverage receive higher weights because MITRE or CTI visibility has limited practical value if the telemetry and detection chain does not close.

Active-weight normalization prevents an out-of-scope metric from artificially lowering or raising the score. Weights therefore express methodological priority while still adapting to the scenario scope.

Validation checks

SACI outputs are supported by consistency checks. These checks ensure that the final score does not contradict graph relations, edge arithmetic remains consistent and N/A handling is applied correctly.

  • Formula check: derived tables, active metric scores and weights must support the final arithmetic.
  • Edge arithmetic: observed edge count plus missing edge count must equal total edge count.
  • Closure check: observed=0 edge count and undeclared endpoints must be reported separately.
  • N/A check: out-of-scope metrics must be removed from the active-weight denominator.
  • Explanation check: the explanation layer must not contradict deterministic outputs.

Short glossary

Declared scope
The defined set of assets, log sources, controls, MITRE techniques and CTI relations included in the evaluation.
Expected edge
A visibility relation that the model expects to close in the graph.
Observed edge
An expected relation that is validated by evidence.
Missing edge
An expected relation that could not be validated by evidence.
Graph closure
The state in which expected relations in the declared scope become observed and no missing relation remains.
Reason code
A label that explains which visibility layer caused a missing relation.
Freshness
The operational recency of telemetry beyond the fact that it was once observed.
Explanation layer
A non-scoring layer that interprets deterministic SACI outputs in natural language.

Integrity-validation experiments

The F-01 narrative is not limited to the historical 97/99 pre-release case. Integrity-gate behavior was tested through nine controlled fault-injection cases, all of which matched their pre-declared expected status. The canonical 99-node / 171-relation structure was also replicated at 1×, 10×, 100×, and 1000× synthetic cardinalities to measure validation time and traced Python peak memory.

The largest synthetic case contained 99,000 nodes and 171,000 edge rows, with a median validation time of 883.438 ms and median peak memory of 41.170 MiB. This result concerns validator data-structure scalability only.
